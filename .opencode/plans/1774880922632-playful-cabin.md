# Website Redesign Plan - One-Time Purchase Product Positioning

## Goal
Redesign `apps/web` so the site converts like premium one-time desktop products (Screen Studio-style polish), and migrate web styling to Tailwind CSS while preserving this product's core truths:
- local/private processing
- offline-after-activation behavior
- public downloads + license-gated runtime
- App / CLI / App+CLI offer model (bundle requires two active keys for desktop processing)

Tailwind migration target:
- move `apps/web` from custom global class CSS to Tailwind utility/component patterns
- keep existing visual brand direction (green/privacy/local) during the migration

shadcn target:
- adopt shadcn/ui primitives for repeated UI patterns (button/card/badge/accordion/etc.)
- keep page-level marketing composition in local components while using shadcn for consistency/accessibility

## Pattern Synthesis to Apply
1. Media-first hero: show product outcome immediately (before/after or workflow proof), then explain.
2. Pricing clarity: explicit "buy once/use forever" language and separate update-window wording.
3. Objection handling near CTAs: refund/update/license transfer/seat policy beside purchase actions.
4. Trust stacked early: logos/testimonials/usage proof and compatibility proof near hero + pricing.
5. Trial/download confidence: reduce friction with clear "download now, activate when ready" flow.
6. Sticky conversion affordance: persistent path to `Pricing` and `Download` on long pages.
7. FAQ as conversion layer: answer practical buying questions (not only technical licensing details).

## Recommended Redesign Direction
### Style direction
- Keep current green/privacy tone but elevate contrast, hierarchy, and product proof density.
- Add richer visual rhythm (media blocks, proof strips, section alternation) without introducing visual noise.
- Use subtle motion (section reveal, card hover, sticky CTA transition), not decorative animation.

### Content direction
- Lead with outcomes and confidence:
  - "remove backgrounds fast"
  - "processed locally"
  - "owned via one-time purchase"
- Clarify App vs CLI vs Bundle with stronger use-case copy and comparison depth.
- Add risk-reversal and policy transparency (only with approved, factual policy wording).

## Tailwind Migration Strategy (apps/web)
### Approach
Use an **incremental migration with a bridge layer**, not a hard cut:
1. Add Tailwind tooling and compile path first.
2. Introduce Tailwind-based base styles/tokens while preserving current behavior.
3. Migrate route by route (`/` -> `/pricing` -> `/downloads` -> `/thank-you`).
4. Remove obsolete legacy CSS classes only after each route is fully converted.

### Why this approach
- Reduces risk from global reset/preflight changes.
- Avoids one large rewrite across all pages at once.
- Keeps checkout/download flows stable while redesign and style migration happen together.

### Tailwind setup plan
- Add Tailwind dependencies to `apps/web/package.json`.
- Add PostCSS wiring file at `apps/web/postcss.config.mjs`.
- Update `apps/web/src/app/globals.css` to Tailwind entry + base token layer.
- Keep CSS variables for brand tokens; expose them through Tailwind-friendly utility usage.

### Migration guardrails
- Eliminate collision-prone legacy class names early (`container` is highest priority).
- Keep `opengraph-image.tsx` and `twitter-image.tsx` on inline styles unless needed.
- Prefer reusable component wrappers for repeated patterns (buttons/cards/sections) to prevent utility sprawl.

## shadcn/UI Integration Strategy (apps/web)
### Setup plan
1. Initialize shadcn in `apps/web` and generate `components.json` with:
   - App Router + TypeScript assumptions
   - `src` aliases (`@/components`, `@/lib/utils`)
   - `globals.css` as style entry
2. Add required runtime helpers/deps used by generated components.
3. Generate first-pass primitives in `src/components/ui`:
   - `button`, `badge`, `card`, `accordion`, `tabs`, `separator`, `alert`, `table`

### Component usage map by route
- `/`: `Button`, `Badge`, `Card`, `Tabs`, `Separator`
- `/pricing`: `Card`, `Table`, `Accordion`, `Button`, `Badge`, `Alert`
- `/downloads`: `Card`, `Tabs`, `Button`, `Alert`, `Separator`
- `/thank-you`: `Alert`, `Card`, `Button`, `Badge`

### Token and theming rules
- Preserve brand semantics by mapping existing CSS vars into shadcn-style semantic tokens (`--background`, `--foreground`, `--primary`, `--muted`, `--border`, `--radius`).
- Do not introduce a dark-mode project in this pass.
- Avoid hardcoded ad-hoc colors in components; keep tokens centralized in `globals.css`.
- Keep legal/licensing meaning stable while visual components change.

## Execution Sequence
1. Tailwind foundation
   - install deps, add PostCSS/Tailwind entry, verify app still renders.
2. shadcn foundation
   - initialize shadcn, add base primitives, add `cn` helper and alias-safe wiring.
3. Shell + token migration
   - migrate `layout.tsx` + `globals.css` to tokenized Tailwind baseline.
4. Route conversion + redesign
   - homepage first, then pricing, downloads, thank-you.
5. Cleanup pass
   - remove unused legacy CSS rules and dead classnames.
6. SEO and copy consistency pass
   - ensure metadata and conversion copy remain aligned after refactor.

## Verification Plan

### Functional and visual verification
1. Install/update deps: `bun install`.
2. Run web locally: `bun run dev:web`.
3. Verify route behavior and links:
   - `/` -> `/pricing` -> checkout start
   - `/` -> `/downloads`
   - purchase success path to `/thank-you`
4. Verify responsive layout (mobile/tablet/desktop) and sticky CTA behavior.
5. Verify copy consistency for:
   - public downloads vs runtime gating
   - App/CLI/Bundle key requirements
   - offline-after-activation promise

### Technical checks
1. Build web app: `cd apps/web && bun run build`.
2. Validate Tailwind is active in production build (utilities emitted, no missing style regressions).
3. Validate shadcn primitives render with expected tokens (no mixed legacy/shadcn style conflicts).
4. Validate metadata output (title/description/OG per route).
5. Confirm no regressions in checkout-triggering UI and API request payloads.

### Final QA checklist
- CTA labels and destinations are consistent across all pages.
- Pricing/policy blocks do not introduce unapproved legal promises.
- FAQ answers match actual product behavior and support flows.
- Motion and section transitions are smooth and non-blocking on low-end devices.
- No stale legacy CSS selectors remain for migrated sections.
