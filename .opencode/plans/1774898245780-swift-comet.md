# Plan: Vercel Analytics + Vercel Flags A/B Testing (apps/web)

## Goal
Implement Vercel Web Analytics and Vercel Flags in `apps/web`, then launch an initial wave of copy experiments focused on improving **checkout completions**.

## Decisions Locked In
- Primary KPI: checkout completions (proxied via thank-you page + checkout context)
- Initial scope: 5-7 high-impact copy tests
- Attribution depth: Vercel-only for now (no DB/event warehouse)

## Implementation Approach

### 1) Add analytics and flags foundations
1. Add dependencies in `apps/web/package.json`:
   - `@vercel/analytics`
   - `flags`
   - `@flags-sdk/vercel`
2. Mount Analytics globally in `apps/web/src/app/layout.tsx`:
   - Add `<Analytics />` from `@vercel/analytics/next` once in root layout.
3. Create centralized flag definitions in `apps/web/src/lib/experiments/flags.ts`:
   - Define all experiments as typed string options using `flag()` + `vercelAdapter()`.
   - Group by page/surface for maintainability.
4. Add discovery endpoint for Vercel Toolbar/Flags Explorer:
   - `apps/web/src/app/.well-known/vercel/flags/route.ts`
   - Use `createFlagsDiscoveryEndpoint` + `getProviderData` and export all flag definitions.
5. Add reusable experiment typing/context helpers:
   - `apps/web/src/lib/experiments/types.ts` (experiment keys, variant unions, context shape)
   - `apps/web/src/lib/experiments/attribution.ts` (compact serialization/parsing for URL payload)

### 2) Instrument event tracking (client + server)
1. Add a small client tracker helper in `apps/web/src/lib/analytics/events.ts`:
   - Wrap `track()` from `@vercel/analytics` with stable event names and typed payloads.
2. Add server tracker where needed (checkout route) using `@vercel/analytics/server`.
3. Event names and required payloads:
   - `experiment_exposure`
     - `experiment_key`, `variant`, `page`, `slot`
   - `checkout_started`
     - `kind`, `page`, `exp`
   - `checkout_completed_proxy`
     - `kind`, `page`, `exp`, `has_exp`
4. Add exposure dedupe component:
   - `apps/web/src/components/analytics/ExperimentExposureTracker.tsx`
   - Fires exposure once per experiment+variant+page per session via `sessionStorage` guard.
5. Add thank-you conversion tracker:
   - `apps/web/src/components/analytics/ThankYouTracker.tsx`
   - Fires `checkout_completed_proxy` when `/thank-you` renders.

### 3) Launch first 6 copy experiments (high-impact)
Evaluate flags in server components and pass variant text to client components when needed.

1. Home hero headline (`apps/web/src/app/page.tsx`)
   - Variants: control vs two outcome/privacy alternatives.
2. Home primary CTA label (`apps/web/src/app/page.tsx`)
   - Variants for `/downloads` CTA copy.
3. Pricing hero headline/subcopy (`apps/web/src/app/pricing/page.tsx`)
   - Variants framing flexibility vs one-time value.
4. Pricing plan button labels (`apps/web/src/app/pricing/PricingClient.tsx` + `pricing/page.tsx`)
   - Variant CTA strings for App/CLI/Bundle plan actions.
5. Sticky CTA labels (`apps/web/src/components/marketing/StickyCta.tsx` and usage in pages)
   - Variant text for primary/secondary CTA labels.
6. Downloads hero headline/subcopy (`apps/web/src/app/downloads/page.tsx`)
   - Variants around install-first and activation clarity.

## Attribution Strategy (No Persistence)
1. Build compact experiment context token `exp` (e.g., serialized key:variant pairs).
2. Propagate `exp` through CTA links and checkout initiation payload.
3. Update `apps/web/src/app/api/checkout/route.ts`:
   - Accept `exp` in POST body.
   - Append `exp` and `kind` to `success_url` (`/thank-you?exp=...&kind=...`).
   - Emit server event `checkout_started` (with `kind` + `exp`).
4. Update `apps/web/src/app/thank-you/page.tsx`:
   - Read `searchParams` for `exp` and `kind`.
   - Render `ThankYouTracker` to emit `checkout_completed_proxy`.

## Verification Plan

### Local checks
1. Install deps (workspace): `bun install`
2. Build web app: `cd apps/web && bun run build`
3. Run dev server: `cd apps/web && bun run dev`
4. Manual validation:
   - `/`, `/pricing`, `/downloads` render with expected variant text.
   - Exposure events fire once per experiment per session.
   - Pricing CTA click sends `checkout_started` with `exp` + `kind`.
   - `success_url` includes `exp` and `kind`.
   - `/thank-you` emits `checkout_completed_proxy` with `exp` context.

### Vercel/Flags validation
1. Create flags in Vercel Flags dashboard (matching keys in code).
2. Pull env vars locally if needed: `vercel env pull`.
3. Confirm discovery endpoint works:
   - `/.well-known/vercel/flags`
4. In preview, use Flags Explorer overrides to verify each variant maps to correct copy.
5. In Vercel Analytics dashboard:
   - Verify events: `experiment_exposure`, `checkout_started`, `checkout_completed_proxy`
   - Filter by experiment properties to compare variant lift.

## Risks and Mitigations
- Multiple simultaneous tests can confound results.
  - Mitigation: run all 6 but prioritize interpretation by funnel stage and keep naming clear.
- Thank-you is a proxy for completion, not authoritative purchase confirmation.
  - Mitigation: treat as proxy KPI now; optionally add Polar webhook correlation in later phase.
- Missing `FLAGS` env setup can prevent managed flag evaluation.
  - Mitigation: document dashboard + `vercel env pull` setup in rollout notes.
- Client-side event blocking (ad/privacy tools) may reduce absolute counts.
  - Mitigation: evaluate relative lift across variants, not absolute event totals.

## Rollout Order
1. Foundation (deps + analytics mount + flags registry + discovery endpoint)
2. Instrumentation (`experiment_exposure`, `checkout_started`, `checkout_completed_proxy`)
3. Copy variants on home/pricing/downloads/sticky surfaces
4. QA in preview with Flags Explorer overrides
5. Production rollout with cautious variant allocations, then iterate based on conversion lift
