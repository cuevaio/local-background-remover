# Plan: Pricing/License V2 Refinement (App, CLI, Both)

## Goal
Refine the current implementation to match the confirmed Polar setup and improve the web UX quality:
- Products: `app`, `cli`, `both` (prices: 6.99, 6.99, 9.99)
- Benefits: `cli`, `app` (currently wired in env as `POLAR_BENEFIT_CLI_ID` and `POLAR_BENEFIT_DESKTOP_ID`)
- `both` customers receive two keys and must activate both:
  - app key in desktop app
  - cli key in CLI

Desktop processing must require both activations; terminal CLI must require CLI activation.

## Last Review: Current Mismatches to Fix
1. Checkout + pricing still use old kinds (`bundle`, `upgrade`).
2. Web copy still says one license unlocks both and includes upgrade messaging.
3. Token issuance currently gives desktop activation implicit CLI surface.
4. CLI license cache is single-token (cannot represent separate app+cli activations).
5. Desktop gate checks only desktop status before processing.
6. Root/env/docs still reference old product env names.

## Recommended Architecture Updates

### 1) Entitlement model and naming
- Keep runtime surfaces as `cli` and `desktop` in code (to match existing CLI/desktop calls and env naming), while product copy says `app`.
- Map product kinds to checkout kinds:
  - `cli` -> `POLAR_PRODUCT_CLI_ID`
  - `app` -> `POLAR_PRODUCT_APP_ID`
  - `both` -> `POLAR_PRODUCT_BOTH_ID`
- Benefit checks:
  - CLI activation/usage requires `POLAR_BENEFIT_CLI_ID`
  - App/desktop activation/usage requires `POLAR_BENEFIT_DESKTOP_ID`

### 2) Remove implicit cross-surface unlock
- Web token issuer must mint one-surface tokens only:
  - desktop activation -> `surfaces: ["desktop"]`
  - cli activation -> `surfaces: ["cli"]`
- No automatic desktop->cli privilege in token payload.

### 3) Multi-activation local cache schema
Move from one cached token to per-surface records:

```json
{
  "version": 2,
  "activations": {
    "cli": { "license_key": "...", "token": "...", "activated_at": 0 },
    "desktop": { "license_key": "...", "token": "...", "activated_at": 0 }
  }
}
```

Rules:
- `license activate --surface cli` updates `activations.cli` only.
- `license activate --surface desktop` updates `activations.desktop` only.
- `license status --surface X` reads only that activation.
- Backward migration: read legacy single-token file and migrate on first write.

### 4) Desktop->CLI runtime enforcement
- Desktop processing operations (`ensure-model`, `remove-background`) must require both statuses valid:
  - desktop active
  - cli active
- CLI gets a new enforcement flag for combined checks in app-launched workflows:
  - `--require-surface` (repeatable)
- Desktop invokes processing as:
  - `--surface cli --require-surface desktop`

This guarantees app-triggered CLI work only when both keys are activated.

## Implementation Plan

### Phase A - Product/env remap and API correctness
1. Update web checkout kind enum to `app|cli|both`.
2. Update product ID mapping to the confirmed env vars.
3. Update API validation/error copy to remove bundle/upgrade references.
4. Update root env/docs to only the confirmed env var set.

### Phase B - License engine hardening (web + CLI)
1. Change token issuance to single-surface only.
2. Refactor CLI cache to multi-activation schema.
3. Add migration logic from old single-token state.
4. Extend CLI enforcement to support required secondary surface checks.

### Phase C - Desktop dual-activation UX and gating
1. Replace single license panel with two key flows (App key, CLI key).
2. Add dual status badges and combined readiness state.
3. Gate model ensure/remove on combined readiness.
4. Improve error messaging for missing specific key.

### Phase D - Landing/pricing redesign ("give it love")
1. Redesign landing hierarchy:
   - hero with stronger visual + headline
   - trust strip
   - 3-step activation flow
   - product comparison
   - testimonials + FAQ + final CTA
2. Redesign pricing page:
   - cards only for App, CLI, Both
   - explicit "Both includes 2 keys" messaging
   - no upgrade language anywhere
3. Update downloads + thank-you pages with precise two-key activation instructions.

### Runtime scenarios
1. **CLI key only active**
   - terminal CLI works
   - desktop processing blocked (missing app key)
2. **App key only active**
   - desktop opens but processing blocked (missing CLI key)
   - terminal CLI blocked
3. **Both keys active**
   - desktop ensure/remove works
   - terminal CLI works
4. **Wrong key type per surface**
   - activation fails with entitlement mismatch
5. **Offline behavior**
   - each surface obeys active + grace windows independently
