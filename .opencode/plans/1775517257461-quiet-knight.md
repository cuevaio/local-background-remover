# Measurement-First Plan For Ads, SEO, And Experiments

## Recommendation
- The current number of concurrent experiments is too high for a likely low-volume paid + organic funnel.
- Treat measurement as the first product change, then reduce active tests to 1 primary experiment at a time.
- Use a minimal attribution spine from landing visit to verified Polar purchase so Ads, SEO, and onsite experiments can be evaluated against the same conversion path.

## Why The Current A/B Test Load Does Not Make Sense
- `apps/web/src/lib/experiments/types.ts` defines 18 assignment dimensions across home, pricing, downloads, compare, docs, and gallery.
- `apps/web/src/app/page.tsx` tracks 12 homepage exposure events in one render path.
- `apps/web/src/app/pricing/page.tsx` tracks 3 more pricing exposures.
- For a new Google Ads campaign and a likely modest traffic site, purchases will be too sparse to support many simultaneous tests.
- Even if flags are analyzed independently, overlapping copy changes increase noise, interaction effects, and time-to-learning.

## Recommended Operating Model
- Run 1 active primary experiment on `/pricing`.
- Optionally run 1 supporting experiment on `/` only if traffic justifies it.
- Default to 2-way tests instead of 3-way tests unless volume is clearly high.
- Optimize experiments on `checkout_started` and verified purchase, not only CTA clicks.
- Keep SEO pages semantically stable; avoid testing titles/canonicals/schema on indexed pages.

## Measurement Architecture

### Phase 1: Capture Attribution On Landing
- Extend `apps/web/src/middleware.ts` to persist a compact first-party attribution cookie.
- Store:
  - `utm_source`
  - `utm_medium`
  - `utm_campaign`
  - `utm_content`
  - `utm_term`
  - `gclid`
  - `gbraid`
  - `wbraid`
  - `landing_path`
  - `referrer_host`
  - `first_seen_at`
  - `last_seen_at`
  - `exp`
  - existing `rmbg_visitor_id` linkage
- Keep `exp` URL propagation for now, but do not rely on UTM persistence in URLs after landing.

### Phase 2: Read Attribution In Checkout
- Add a new helper such as `apps/web/src/lib/analytics/attribution.ts` to parse, merge, serialize, and read the attribution cookie.
- Update `apps/web/src/app/api/checkout/route.ts` to:
  - read the attribution cookie server-side
  - attach a compact attribution snapshot to server analytics
  - attach attribution metadata to Polar checkout if supported
  - pass a minimal subset into the success URL for `/thank-you`

### Phase 3: Fire Explicit Conversion On Thank-You
- Update `apps/web/src/app/thank-you/page.tsx` and `apps/web/src/components/analytics/ThankYouTracker.tsx` to read attribution context.
- Add one explicit Google Ads conversion event on the thank-you page in addition to the existing proxy analytics event.
- Keep the thank-you page noindexed.

### Phase 4: Make Polar Webhook The Purchase Source Of Truth
- Upgrade `apps/web/src/app/api/polar/webhook/route.ts` from logging-only to structured purchase tracking.
- Parse successful purchase events and log a definitive server-side purchase event with:
  - product/kind
  - amount/currency
  - checkout or order identifiers
  - carried attribution metadata
  - experiment token if present
- This becomes the canonical purchase record; thank-you remains the fast client-side signal.

## Keep Ads, SEO, And Experiments Aligned
- Use one shared intent map for both paid and organic landing pages.
- For each core paid theme, define:
  - query cluster
  - primary landing page
  - main promise
  - primary CTA
  - proof block
- Keep ad themes and SEO page messaging aligned on the same high-intent topics:
  - offline background remover for Mac
  - private/local processing
  - batch background removal
  - product photo cleanup
- Use `utm_content` for ad creative identity and `exp` for onsite variant identity; do not merge them.
- Use verified purchase and checkout start as the shared decision metrics across Ads and experiments.
