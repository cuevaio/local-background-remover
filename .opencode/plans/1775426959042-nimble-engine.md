# Add `rmbg upgrade` Command

## Goal
Add a macOS-only `rmbg upgrade` command to the CLI that:
- checks the latest released CLI version before doing any install work
- skips when the current CLI version already matches the latest release
- otherwise reuses the existing public installer flow to install the managed binary at `~/.local/bin/rmbg`
- still installs to the managed path even when the currently running executable came from somewhere else

## Recommended Approach
1. Add a new top-level `upgrade` subcommand in `apps/rmbg/rmbg_cli/cli.py`.
2. Keep the command handler thin in `cli.py` and move the upgrade behavior into a new module: `apps/rmbg/rmbg_cli/upgrade.py`.
3. Reuse the existing web installer contract instead of trying to replace the running executable in place.
4. Compare versions by normalized equality only: current CLI version vs latest release tag with one leading `v` stripped from both.
5. If versions match, return success and report that no upgrade was needed.
6. If versions differ, fetch the installer script from `https://local.backgroundrm.com/install`, execute it with `RMBG_VERSION=<latest-tag>`, and verify `~/.local/bin/rmbg --version` afterward.

## Why This Shape
- The repo already has a stable public installer flow in `apps/web/src/app/install/route.ts` and a matching repo script in `scripts/install.sh`.
- The desktop app already relies on that installer pattern for runtime sync, so the CLI should align with the same managed install layout.
- Installing to `~/.local/bin/rmbg` avoids brittle self-overwrite behavior and handles non-managed current executables cleanly.
- Equality comparison is enough for the user requirement and avoids unnecessary custom semver logic.

## Files To Modify
- `apps/rmbg/rmbg_cli/cli.py`
- `apps/rmbg/rmbg_cli/upgrade.py` (new)
- `apps/rmbg/tests/test_cli_upgrade.py` (new)

## Implementation Details

### 1. CLI Surface
In `apps/rmbg/rmbg_cli/cli.py`:
- register `upgrade` as a top-level subcommand
- add output flags via existing `_add_output_flags()`
- add the handler with `set_defaults(func=cmd_upgrade)`
- extend the parser epilog with an `rmbg upgrade` example

Recommended flags:
- `--force`: run installer even if current version already matches latest
- `--metadata-url`: default `https://local.backgroundrm.com/api/releases/latest`
- `--install-url`: default `https://local.backgroundrm.com/install`

Output behavior should match existing CLI conventions:
- use `_print(...)`
- support text and JSON modes
- return `0` on skip or success
- return `1` for upgrade-related failures

### 2. Upgrade Module
Create `apps/rmbg/rmbg_cli/upgrade.py` for the non-parser logic.

Recommended responsibilities:
- normalize versions/tags
- fetch latest release metadata
- fetch the installer script
- execute the installer via `/bin/bash`
- verify the managed install afterward
- return a structured result payload that `cli.py` can print

Prefer stdlib and existing repo patterns:
- `urllib.request` for HTTP fetches
- existing TLS helper if available/suitable
- `subprocess.run(..., check=False)` for installer execution and post-install version verification
- `Path.home() / ".local/bin/rmbg"` as the managed executable path

### 3. Version Resolution
Use two values:
- current version: existing `RMBG_VERSION` from `cli.py`
- latest version: `GET /api/releases/latest` from `apps/web/src/app/api/releases/latest/route.ts`

Normalization rule:
- trim whitespace
- strip one leading `v`
- compare normalized values for equality

Treat these as errors:
- metadata fetch failure
- invalid JSON
- `ok: false`
- missing or empty `tag`

Do not implement version ordering logic unless execution reveals a concrete need.

### 4. Installer Execution
Re-use the existing install flow rather than reconstructing archive download logic in Python.

Execution model:
- fetch `https://local.backgroundrm.com/install`
- run the fetched script with `/bin/bash`
- set `RMBG_VERSION=<latest-tag>` in the subprocess environment so the install step is pinned to the version already checked

This keeps the installer responsible for:
- macOS target detection
- archive selection
- checksum verification
- extraction to `~/.local/share/rmbg/<tag>/<target>`
- symlink management for `~/.local/bin/rmbg`

### 5. Non-Managed Current Executables
If the command is run from a binary outside `~/.local/bin/rmbg`:
- do not try to overwrite the running binary
- still install the managed binary to `~/.local/bin/rmbg`
- report both the current executable path and the managed executable path in the success payload

Skip behavior remains version-based:
- if current version already equals latest version, skip even if current executable is not the managed path
- `--force` overrides the skip

### 6. Result Payloads
Skip case:
- text: `rmbg <current> is already up to date.`
- JSON fields: `ok`, `upgraded=false`, `current_version`, `latest_version`, `source`

Success case:
- text: `Upgraded rmbg to <latest> and installed managed binary at ~/.local/bin/rmbg.`
- JSON fields: `ok`, `upgraded=true`, `current_version`, `latest_version`, `current_executable`, `managed_executable`, `source`

Failure case:
- concise error surfaced from metadata fetch, installer fetch, installer subprocess failure, unsupported platform, or post-install verification mismatch

## Edge Cases To Handle
- current version already equals latest
- `--force` with matching versions
- latest endpoint returns `ok: false`
- latest endpoint returns malformed JSON or missing `tag`
- installer fetch fails
- installer subprocess exits nonzero
- post-install `~/.local/bin/rmbg --version` does not match target
- command run from a non-managed executable path
- command run on non-macOS should fail clearly and early
