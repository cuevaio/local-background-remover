# Add Copy Image Button For Processed Results

## Recommended Approach
- Implement this in `apps/desktop` only. The real processed-result UI exists there today; `apps/web` only has marketing examples.
- Add a `Copy image` action to the existing result action row in the desktop focus panel.
- Keep the renderer sandboxed and route clipboard writes through preload + IPC.
- Copy the actual processed bitmap image from `item.output_path`, not the file path text.

## Files To Modify
- `apps/desktop/src/renderer/index.html`
- `apps/desktop/src/renderer/renderer.ts`
- `apps/desktop/src/preload/preload.ts`
- `apps/desktop/src/renderer/global.d.ts`
- `apps/desktop/src/main/main.ts`
- `apps/desktop/src/renderer/styles.css` only if button spacing needs a small adjustment

## Implementation Plan
1. Update `apps/desktop/src/renderer/index.html`
- Add a new `Copy image` button in `.focus-actions` near `Save as...` and `Open folder`.

2. Update `apps/desktop/src/renderer/renderer.ts`
- Add a DOM reference for the new button.
- In `renderFocusPanel()`, hide the button when there is no `item.output_path`, matching the existing processed-only actions.
- Add a click handler that:
- reads `currentItem()`
- returns early if there is no processed output
- disables the button while the request is running
- calls a new preload method such as `window.rmbg.copyProcessedImage({ sourcePath: item.output_path })`
- shows a success toast/status like `Copied image to clipboard.`
- shows an error toast/status if the copy fails

3. Update `apps/desktop/src/preload/preload.ts`
- Add a small payload type for the copy request.
- Expose a new bridge method, e.g. `copyProcessedImage`, using `ipcRenderer.invoke("copy-processed-image", payload)`.

4. Update `apps/desktop/src/renderer/global.d.ts`
- Add the renderer typing for the new `window.rmbg.copyProcessedImage(...)` API and its response shape.

5. Update `apps/desktop/src/main/main.ts`
- Add a new IPC handler for `copy-processed-image`.
- Use existing path validation helpers such as `normalizeUserPath(...)`.
- Verify the processed file exists before attempting to copy it.
- Use Electron clipboard APIs to write image data, not text:
- `nativeImage.createFromPath(...)`
- `clipboard.writeImage(...)`
- Throw clear errors for missing or unreadable output files.

## Constraints And Notes
- Follow `apps/desktop/AGENTS.md`: keep renderer sandboxed and expose only a minimal preload API.
- Treat the incoming path as untrusted in main, even though it comes from library state.
- Keep the before/after workflow unchanged; this is an additional result action only.
- Reuse existing button styling unless spacing forces a tiny CSS tweak.

## Verification
1. Run the desktop app with `cd apps/desktop && bun run dev`.
2. Activate the app if required and import an image.
3. Wait for processing to finish and confirm `Copy image` only appears for processed items.
4. Click `Copy image` and verify the app reports success.
5. Paste into another app that accepts pasted images and confirm the processed image appears.
6. Round-trip check: paste back into the desktop app and confirm the existing clipboard-image import flow treats it as an image.
7. Negative check: select an unprocessed item and confirm the button is hidden or unavailable.
8. Negative check: if the processed output file is missing, confirm the app shows a clear error instead of silently failing.

## Expected Scope
- Small desktop-only feature.
- No web changes.
- No new dependencies.
