# Ship HTTPS TLS Fix To Global CLI Binary

## Goal
- Make `rmbg remove --input https://...` work from any directory, including `~/`, not just when running the editable `apps/rmbg` project.
- Ship the already-implemented TLS fix in the globally installed `rmbg` binary that lives under `~/.local/bin`.
- Keep the release scope minimal: version bump, release, and verification of the installed binary.

## Root Cause
- The source fix in `apps/rmbg` is working: `uv run --project apps/rmbg rmbg ...` succeeds.
- The failing command from `~/` is using the installed binary at `~/.local/bin/rmbg`, which points to a released PyInstaller artifact under `~/.local/share/rmbg/...`.
- That installed binary still contains the old code, so users outside the repo do not benefit until a new CLI release is built, published, and reinstalled.

## Recommended Changes
1. Bump the CLI version in `apps/rmbg/pyproject.toml` to the next patch release.
2. Keep the existing TLS source changes in `apps/rmbg` as the payload for the release; no further runtime code changes are required for this issue.
3. Do not change PyInstaller or release scripts unless the release build proves they are missing the updated code.
4. Create a new git tag matching the bumped CLI version so `.github/workflows/release.yml` builds and publishes the new archive.
5. Ensure the latest release resolution used by `https://local.backgroundrm.com/install` points at the new tag.
6. Reinstall the global binary from the public installer and verify `rmbg` works from `~/`.

## Files To Modify
- `apps/rmbg/pyproject.toml`
- `.opencode/plans/1775412684283-calm-island.md`
- Optional release note or deployment note only if needed during execution

## Release Steps
1. Update `apps/rmbg/pyproject.toml` version to the next patch release.
2. Run targeted verification in `apps/rmbg` to confirm the released source is green.
3. Commit the version bump together with the TLS fix and this approved plan file.
4. Create and push a matching git tag such as `vX.Y.Z`.
5. Let `.github/workflows/release.yml` publish the CLI tarball and `checksums.txt`.
6. Confirm the GitHub release contains the expected asset name for the current platform.
7. Confirm `https://local.backgroundrm.com/api/releases/latest` resolves to the new tag.
8. Reinstall from `https://local.backgroundrm.com/install` so `~/.local/bin/rmbg` points at the new release directory.

## Verification
- Source verification before release:
  - `cd apps/rmbg && uv run --group dev pytest tests/test_license_manager.py -q`
  - `cd apps/rmbg && uv run --group dev ruff check rmbg_cli tests`
- Installed binary verification after release:
  - `cd ~`
  - `curl -fsSL https://local.backgroundrm.com/install | RMBG_VERSION=vX.Y.Z bash`
  - `command -v rmbg`
  - `rmbg --version`
  - `rmbg remove --input https://www.cueva.io/favicon.ico --output /tmp/cueva-favicon-rmbg.png --json`
- Latest-tag verification:
  - `curl -fsSL https://local.backgroundrm.com/api/releases/latest`
  - Re-run the installer unpinned and confirm it still installs `vX.Y.Z`.

## Risks And Assumptions
- Assumes the current `apps/rmbg` code already fully fixes the TLS issue in packaged builds.
- Assumes `certifi` is already bundled by the current PyInstaller flow; exploration indicates it is.
- Existing users will not get the fix until they reinstall or otherwise refresh the installed binary.
- The latest-release endpoint and installer may be cached briefly, so unpinned installs may lag for a few minutes after release.
