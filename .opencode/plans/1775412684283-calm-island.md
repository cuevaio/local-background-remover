# Prevent Invalid License Activations From Consuming Slots

## Goal
- Fix the license activation backend so wrong-surface or otherwise invalid key attempts are rejected before any activation is created.
- Preserve current happy-path activation behavior for valid keys.
- Add backend test coverage for the activation route, which currently has none.

## Root Cause
- `apps/web/src/app/api/license/activate/route.ts` currently calls Polar activation before it validates entitlement for the requested surface.
- For example, using an APP key with `surface: "cli"` can create a Polar activation first, then fail entitlement validation afterward.
- On retry, Polar may report an activation-limit error because the invalid first attempt already consumed activation state.

## Recommended Changes
1. Update `apps/web/src/app/api/license/activate/route.ts` to preflight with `validateLicenseKey()` before calling `activateLicenseKey()`.
2. After normalizing `surface` and resolving `requiredBenefitId`:
   - call `validateLicenseKey({ key, organizationId })`
   - reject immediately if `license.status !== "granted"`
   - reject immediately if `license.benefit_id !== requiredBenefitId`
3. Only call `activateLicenseKey()` after the preflight validation passes.
4. Keep the existing post-activation `validateLicenseKey(...)` call and offline token issuance logic unchanged.
5. Preserve current error text for entitlement mismatch so CLI behavior stays stable.

## Test Plan
1. Add focused tests for `apps/web/src/app/api/license/activate/route.ts` by mocking `@/lib/polar` and `@/lib/license-token`.
2. Cover these cases:
   - wrong-surface entitlement returns `403` and does not call `activateLicenseKey()`
   - invalid or non-granted key returns failure and does not call `activateLicenseKey()`
   - happy path still performs preflight validate, activation, post-activation validate, and token issuance
   - optional: activation-limit errors still map through the current error helper when activation is actually reached
3. Update `apps/web/package.json` test script only if the current script cannot run the new route tests.

## Verification
- `cd apps/web && bun test src/app/api/license/activate/route.test.ts`
- `cd apps/web && bun run test` if the test script is updated
- `cd apps/web && bun run build`
- Manual repro after implementation:
  - attempt activation with an APP key and `surface: "cli"`
  - expect repeated attempts to keep returning entitlement failure
  - expect no activation-limit regression for the same invalid attempt sequence
  - sanity-check a valid CLI key with `surface: "cli"` still succeeds

## Risks And Assumptions
- This adds one extra Polar `validate` call before successful activations.
- The plan assumes `validateLicenseKey()` without `activationId` returns enough information to check `status` and `benefit_id`; existing code already depends on that behavior elsewhere.
- The safest minimal fix is isolated to the activation route; no Polar helper or refresh-route changes should be needed.
