# Add A/B Testing To All Landing-Page Headings

## Goal
Add per-section A/B tests for all visible heading-like copy on the homepage landing page, using the existing Vercel Flags experiment infrastructure in `apps/web`.

## Confirmed Scope
- Include all visible headings on the landing page, not just major section titles.
- Use per-section experiments, not a single page-wide variant.
- Footer heading experiments should be homepage-only, not global across the site.

## Recommended Approach
1. Keep experiment evaluation centralized in `src/app/page.tsx`.
2. Add one experiment per homepage section or heading bundle, rather than one experiment per individual string.
3. Centralize homepage heading copy into a single typed content module so variants are defined in one place.
4. Pass one structured `copy` prop into each homepage section component instead of scattering variant lookups inside many components.
5. Split the current shared footer so homepage-only heading experiments can live under the home route without affecting other pages.

## Why This Shape
- The repo already has homepage experiment plumbing in `src/lib/experiments/flags.ts`, `src/lib/experiments/types.ts`, `ExperimentExposureTracker`, `ExpLink`, and middleware visitor assignment.
- Current landing-page headings are fragmented across many components and local arrays; centralizing copy is the smallest scalable change.
- Per-section flags keep exposure tracking and attribution manageable while still allowing independent experiments.
- The footer currently lives in `src/app/layout.tsx`, so homepage-only footer testing requires moving the heading-bearing footer content out of the global layout boundary.

## Implementation Steps
1. Inventory homepage heading bundles.
   Define the exact heading surfaces per section so each experiment owns a coherent copy group.
   Expected bundles:
   - hero
   - before/after showcase
   - input options
   - workflow comparison
   - CLI quickstart section
   - automation chats
   - quote section
   - testimonial mosaic
   - pricing/policy/FAQ section
   - sticky CTA
   - homepage footer CTA and footer column headings

2. Add new experiment definitions.
   Extend `src/lib/experiments/types.ts` and `src/lib/experiments/flags.ts` with new homepage section-level assignments and short attribution tokens.
   Keep existing experiments where they already match the desired bundle (`home-hero-headline`, `home-cli-emphasis`, `sticky-cta-copy`) and expand coverage with new section-specific experiments.

3. Create a centralized homepage heading-copy module.
   In `src/content/home/heading-copy.ts`, export typed copy bundles keyed by section and variant.
   This module should hold the heading text currently scattered across page-level constants, component-local arrays, and small content files.

4. Resolve homepage variants in `src/app/page.tsx`.
   Use evaluated assignments to build a `homeHeadingCopy` object for the page.
   Pass section-specific `copy` props into each homepage marketing component.
   Expand `ExperimentExposureTracker` exposures so each section-level experiment is tracked once per page view.

5. Refactor section components to accept copy props.
   Update each homepage component so heading-like strings come from its `copy` prop rather than inline constants.
   For sections with repeated title-like content, pass resolved arrays/objects, not raw variant IDs.

6. Split homepage footer content from the shared layout.
   Move the heading-bearing footer CTA and any footer columns included in homepage testing out of `layout.tsx` and under `page.tsx`.
   Keep a shared non-experiment footer shell only if needed for the rest of the site.
   This ensures homepage footer experiments do not leak to other routes.

7. Keep attribution stable.
   Ensure new homepage experiments still flow through existing `exp` attribution behavior and remain compact enough for internal links.
   Update attribution tests only if token generation or parsing changes.

## Design Constraints
- Prefer minimal structural change beyond what is needed for the footer boundary.
- Do not add a new global provider or context unless page-level prop passing becomes clearly unmanageable.
- Exposure tracking should stay at one event per section experiment, not one event per individual heading string.
- Avoid bloating `exp` tokens with too many micro-experiments; section bundles are the correct granularity.

## Verification
1. Run `bun run build` in `apps/web`.
2. Open `/` and verify all homepage headings render correctly with no hydration or serialization issues.
3. Verify each tested section receives internally consistent copy for its assigned variant.
4. Verify footer heading experiments appear only on the homepage and not on other routes.
5. Verify experiment exposures still emit once per experiment/section and keep the expected `page` and `slot` metadata.
6. Verify internal links still preserve experiment attribution through `ExpLink`/`TrackedExpLink`.
7. Spot check mobile and desktop layouts so longer variant headings do not break section spacing.

## Risks
- Adding too many independent experiments can increase exposure-event volume.
- Attribution tokens can grow if experiment keys are too granular.
- Footer refactor is the main structural risk because it crosses the current layout boundary.
- Variant copy length may cause layout regressions on small screens; mobile validation is required.
