# Plan: External runtime-only desktop releases

## Diagnosis
- The `v0.4.3` release failed because the desktop app embeds the full PyInstaller `rmbg` onedir bundle, and macOS signing choked on the bundled `_internal/Python.framework` inside the Electron app bundle.
- The desktop app embeds Python today because it shells out to the `rmbg` CLI for model bootstrap, inference, and license-gated operations, and the packaged app has been shipping a self-contained fallback runtime at `Contents/Resources/rmbg/rmbg`.
- Packaged desktop already installs and version-matches an external CLI runtime from `https://local.backgroundrm.com/install` into `~/.local/bin/rmbg`, then only falls back to the bundled runtime if the external path is missing or mismatched.
- User decision: remove the bundled CLI/runtime from the desktop app entirely and rely only on the external installed runtime.

## Recommended Implementation
1. Remove bundled-runtime resolution from the desktop main process.
- In `apps/desktop/src/main/rmbg-runtime.ts`, delete packaged-resource lookup (`process.resourcesPath/rmbg/rmbg`) and keep only:
  - `RMBG_DESKTOP_CLI_PATH` override
  - installed runtime path for packaged apps
  - existing dev resolution (`.venv/bin/rmbg`, then `uv run`) for local development
- In `apps/desktop/src/main/main.ts`, keep `ensureDesktopRuntimeInstalled()` as the packaged-app setup path, but remove the fallback to a bundled runtime.
- After install/update, require the installed runtime path to exist and match `app.getVersion()`, otherwise throw a clear runtime-setup error.

2. Make runtime setup a first-class UX state instead of relying on silent fallback.
- In `apps/desktop/src/renderer/renderer.ts`, add clearer user-facing messaging for:
  - installer download failure
  - installer execution failure
  - installed binary missing after install
  - installed binary version mismatch
- Keep the app launchable, but disable runtime-dependent actions until setup succeeds.
- Add an explicit retry path for runtime setup, such as a `Retry runtime install` action that re-runs `ensureDesktopRuntimeInstalled()`.

3. Remove the bundled CLI from desktop packaging.
- Delete `apps/desktop/scripts/prepare-rmbg-runtime.mjs`.
- Remove `build-resources/rmbg` packaging from `apps/desktop/electron-builder.yml`.
- Remove any runtime-specific `mac.binaries` entry tied to the bundled CLI.
- Update `apps/desktop/package.json` so `dist:mac` no longer depends on copying or staging the bundled `rmbg` runtime.

4. Simplify the release workflow around a pure Electron app bundle.
- In `.github/workflows/release.yml`, remove the `Build packaged CLI runtime` step from the `build-desktop` job.
- Keep the separate CLI release artifact job, because the public install script still needs the CLI tarball published to GitHub Releases.
- Keep desktop signing/notarization verification, but it should now operate on an Electron app bundle with no embedded Python runtime.

5. Update docs to reflect the external-runtime contract.
- In `apps/desktop/README.md`, document that packaged desktop auto-installs or updates `rmbg` externally from `https://local.backgroundrm.com/install` and no longer bundles the CLI inside the `.app`.
- Keep `RMBG_DESKTOP_CLI_PATH` documented as a support/developer override.
- In `DEPLOYMENT_RUNBOOK.md`, document that desktop and CLI remain intentionally version-coupled and that the CLI release artifact must be available before the desktop app can self-install the matching runtime.

## Why Python Was Embedded
- PyInstaller packages `rmbg` as a standalone executable plus its Python runtime and native ML dependencies, so embedding it inside the app bundle provided a self-contained fallback that could run without requiring users to install Python, `uv`, PyTorch, or the CLI separately.
- With this plan, the Electron app will stop shipping Python entirely. Runtime ownership moves to the external installed `rmbg` artifact delivered by the public install script.

## Risks To Accept
- First packaged launch now depends on successful network access if the runtime is not already installed.
- Desktop and CLI versions must stay in lockstep, because desktop still expects `rmbg --version === app.getVersion()`.
- Desktop-driven installs will continue to replace the shared user-level `~/.local/bin/rmbg` symlink, which may affect users who also use the CLI directly from Terminal.
- Offline use still works only after a matching runtime is already installed and licensing is active or within grace; offline first use will no longer be supported.

## Verification
1. Run desktop tests that cover runtime resolution and error handling, updating or adding tests so packaged mode no longer expects `Contents/Resources/rmbg/rmbg`.
2. Build a desktop package with `cd apps/desktop && bun run dist:mac` and confirm the app bundle no longer contains `Contents/Resources/rmbg` or an embedded Python runtime.
3. Verify the release workflow no longer stages the bundled CLI in the desktop job, but still publishes the CLI artifact in the dedicated CLI job.
4. Manual packaged-app scenarios on macOS:
   - no installed runtime, network available: desktop auto-installs the matching CLI and proceeds
   - wrong installed version: desktop upgrades/downgrades to the matching CLI version
   - no network and no installed runtime: desktop stays open but shows a clear runtime-setup failure
   - matching runtime already installed: desktop works without reinstall
   - offline with previously installed matching runtime: desktop continues to work subject to existing license grace behavior
