# Fix Website A/B Testing With Flags SDK + Vercel CLI

## Goal
- Restore production-ready A/B testing in `apps/web`.
- Fix the current `control`-only behavior by validating and repairing Vercel Flags setup.
- Preserve experiment attribution across internal marketing navigation so conversions stay tied to the original experiment context.

## Root Cause
- `apps/web/src/lib/experiments/flags.ts` intentionally falls back to `control` when `process.env.FLAGS` is missing, so any local/preview/prod environment without a valid Vercel Flags bundle will never bucket visitors.
- Experiment context is partially implemented: `exp` is created on `/`, `/pricing`, and `/downloads`, but many internal links still drop it before checkout and thank-you conversion tracking.
- The repo does not currently document or script the required Vercel Flags CLI workflow, so the dashboard/env state can drift from the code-defined flags.

## Recommended Implementation
1. Verify and repair Vercel Flags project state with the Vercel CLI.
   - Confirm the repo is linked to the correct Vercel project.
   - Inspect existing remote flags and compare them to the code-defined keys/options in `apps/web/src/lib/experiments/flags.ts`.
   - Create missing flags, update mismatched metadata/options where supported, and remove stale flags only if clearly safe.
   - Pull fresh env vars so `FLAGS` and `FLAGS_SECRET` are present for local/dev validation.

2. Keep the existing flag declaration architecture and make it operational.
   - Reuse `apps/web/src/lib/experiments/flags.ts` as the single source of truth for code-defined flags.
   - If needed, tighten comments or fallback handling there, but do not redesign the flag system.

3. Add one shared internal link helper that preserves `exp` automatically.
   - Create a small client-side wrapper around `next/link` that reads the current `exp` query param and applies `withExpParam()` only to internal app routes.
   - Keep the helper minimal and focused on string hrefs used throughout the marketing site.

4. Replace plain internal marketing links with the new helper across the funnel.
   - Update direct CTAs on:
     - `apps/web/src/app/page.tsx`
     - `apps/web/src/app/pricing/page.tsx`
     - `apps/web/src/app/downloads/page.tsx`
   - Update shared navigation and footer surfaces on:
     - `apps/web/src/app/layout.tsx`
     - `apps/web/src/components/marketing/StickyCta.tsx`
     - `apps/web/src/components/marketing/SiteFooter.tsx`
   - Update other high-traffic marketing components/pages that link deeper into the funnel, especially compare/gallery/docs-related entry points found during implementation.

5. Make the repo ready for ongoing Vercel Flags operations.
   - Add `apps/web` scripts for common workflows such as listing/inspecting flags and pulling env vars.
   - Document the setup and validation flow in `apps/web/README.md` and, if needed, the deployment/runbook docs so production setup is explicit.
   - Ensure docs call out that missing `FLAGS` means all variants resolve to `control`.

## Execution Notes
- Prefer using the existing `flags-sdk` integration instead of introducing middleware/precompute in this pass.
- Preserve current server-side evaluation, exposure tracking, checkout tracking, and thank-you tracking.
- Keep the implementation minimal: one reusable link helper, targeted link replacements, and explicit CLI/docs support.
- Use the Vercel CLI to reconcile remote flags with the code-defined flag set during implementation rather than leaving those steps as follow-up guidance.
